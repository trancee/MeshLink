# Research: destination-sourced sequence numbers, Hello/IHU, and RouteDigest resync for MeshLink's Babel-inspired routing (issue #130)

This is fact-finding only. No design recommendation is made here; the design
decision is deferred to the repo owner in a follow-up step.

## Summary

RFC 8966 defines node-originated sequence numbers that only ever increase via
an explicit seqno-request round trip to the destination, plus Hello/IHU as a
periodic, interval-driven liveness/cost protocol between stable neighbours —
and it has **no** route-table-digest or differential-resync mechanism of any
kind. MeshLink's current implementation does not follow either the RFC's
seqno-origination model (it locally mints/bumps `directRouteSeqNos` on every
BLE reconnect, not on an explicit request from the destination) or a real
Hello/IHU liveness protocol (Hello/IHU wire frames exist but are only
partially wired to interval/history logic based on what's in the codec), and
its `RouteDigest` is a from-scratch, MeshLink-only design (a single 32-bit
FNV-1a hash of the whole route table) with no RFC precedent and, as of this
reading, no actual differential-resync logic behind it — `onRouteDigest` is a
no-op stub in `RouteCoordinator.kt`.

## Findings

### 1. RFC 8966: sequence-number origination and increment (Section 2.5, 3.2.2, 3.7)

1. **A node's own sequence number only increases via explicit request, never spontaneously.** "A node increments its sequence number (modulo 2^16) whenever it receives a request for a new sequence number (Section 3.8.1.2). A node SHOULD NOT increment its sequence number (seqno) spontaneously, since increasing seqnos makes it less likely that other nodes will have feasible alternate routes when their selected routes fail." — RFC 8966 §3.2.2 ("Node Sequence Number").
2. **Babel deliberately rejected DSDV's periodic-seqno-bump model** in favor of on-demand seqno requests precisely to avoid unnecessary invalidation of other nodes' feasibility distances: "In DSDV, the sequence number of a source is increased periodically... Babel takes a different approach. When a node detects that it is suffering from a potentially spurious starvation, it sends an explicit request to the source for a new sequence number." — RFC 8966 §2.6 ("Requests").
3. **When announcing a locally-originated route, the seqno used is "the local router's sequence number"** — i.e. one counter per originating node, reused across all self-originated prefixes/routes, not reset or bumped per route or per reconnect event: "If an update is for a route injected into the Babel domain by the local node... the seqno is set to the local router's sequence number." — RFC 8966 §3.7 ("Sending Updates").
4. **Sequence numbers are explicitly excluded from route-selection preference** — a route is never preferred just because its seqno is newer; seqno only gates *feasibility*, not selection: "Route selection MUST NOT take seqnos into account: a route MUST NOT be preferred just because it carries a higher (more recent) seqno." — RFC 8966 §3.6.
5. This means MeshLink's current `RouteCoordinator.onPeerConnected` behavior — minting a fresh, monotonically-incremented `directRouteSeqNo` for the direct-route entry on *every* BLE reconnect regardless of whether anything about the route actually changed — is a structural departure from RFC 8966's model, where seqno increments are rare, destination-sourced, and reserved for satisfying an explicit starvation-recovery request. See `RouteCoordinator.kt:34-37`: `val seqNo = (directRouteSeqNos[peerId.value] ?: 0L) + 1L; directRouteSeqNos[peerId.value] = seqNo`. This runs unconditionally inside `onPeerConnected`, i.e. on every reconnect.

### 2. RFC 8966: SeqNo request mechanism (Sections 2.6, 3.8.1.2, 3.8.2.1)

6. **Two request types exist**: route requests (never forwarded, simply ask "send me your current update for this prefix") and seqno requests (forwarded hop-by-hop toward the source if the receiver cannot satisfy them locally, and specify a target seqno value). — RFC 8966 §3.8 ("Explicit Requests").
7. **Receiving-node behavior on a seqno request** (§3.8.1.2): if the receiver has a selected finite-metric route for the requested prefix and (a) its router-id differs from the requested router-id, or (b) router-ids match and its stored seqno is already ≥ requested, it **answers directly** with an Update. If router-ids match and the requested seqno is *larger* than what it has, and the requested router-id is *its own*, it increments its own seqno by exactly 1 and answers. "A node MUST NOT increase its sequence number by more than 1 in reaction to a single seqno request." Otherwise (router-id isn't its own and it can't satisfy locally), it decrements the Hop Count and **forwards** the request to the next hop of a feasible (preferentially) or unfeasible route toward that prefix, if the hop count is ≥2 and it is currently advertising that prefix; it also maintains a "recently forwarded" list to dedupe and to route the eventual reply back.
8. **What triggers origination of a seqno request** (§3.8.2.1, "Avoiding Starvation"): "A node that has lost all feasible routes to a given destination but still has unexpired unfeasible routes to that destination MUST send a seqno request; if it doesn't have any such routes, it MAY still send a seqno request." The requested seqno is `(feasibility-distance seqno) + 1`, carrying a hop-count field (RFC suggests 64 as a diameter-safe default) to bound forwarding-loop exposure, since seqno-request forwarding deliberately bypasses the feasibility condition.
9. **Loop safety for request forwarding** relies on hop-count decrement plus duplicate-request detection at each forwarding node (§2.6, §3.8.1.2), not on feasibility — this is a distinct safety mechanism from the route-update feasibility condition.
10. Additional related triggers: a node SHOULD send a *unicast* seqno request when it receives an unfeasible update for a currently-selected route (§3.8.2.2, "Dealing with Unfeasible Updates") or shortly before a selected route's expiry timer would fire, to try to refresh/retract it first via a plain route request (§3.8.2.3, "Preventing Routes from Expiring").
11. MeshLink's current `WireFrame.SeqNoRequest` (per `RoutingPayloadCodec.kt` / `WirePayloadCodecSchema.kt`) carries only `destinationPeerId` and `requestedSeqNo` — there is **no hop-count field**, no forwarding-neighbour bookkeeping, and (per `RouteCoordinator.kt`) no handler for `WireFrame.SeqNoRequest` at all is implemented in the reviewed coordinator; only `onRouteDigest` and the direct route/update/retraction handlers exist. This means the seqno-request half of Babel's starvation-recovery machinery is defined on the wire but not implemented in `RouteCoordinator`, and the wire schema as currently defined **cannot carry a hop count** for loop safety if request-forwarding is later implemented — that would require a wire-format change (a new field), not just new coordinator logic.

### 3. RFC 8966: Hello/IHU neighbour acquisition (Sections 3.4, 3.4.1, 3.4.2, 3.4.3, 4.6.5, 4.6.6)

12. **Neighbour-table entry creation is intentionally implementation-defined** and can be as simple as "creating an entry when any Babel packet is received" (§3.4) — RFC 8966 does not mandate a specific discovery handshake, which gives latitude for a BLE-connection-driven neighbour model instead of a Hello-driven one.
13. **Hello TLV fields** (§4.6.5): `Flags` (bit 0 = Unicast flag), `Seqno` (a per-interface counter for Multicast Hellos, or a per-neighbour counter for Unicast Hellos — **not** the node's route seqno; these are unrelated counters), `Interval` (upper bound in centiseconds until the next Hello of the same kind; if `0`, this Hello is "unscheduled" and doesn't create a new promise/deadline). "We say that a Hello is 'scheduled' if it carries a nonzero interval, and 'unscheduled' otherwise." (§3.4.1).
14. **Multicast vs Unicast Hello semantics** (§3.4.1): Multicast Hello uses one seqno counter per *interface* and must reach all neighbours on that link (via multicast or per-neighbour unicast copies); Unicast Hello uses a separate seqno counter per *neighbour* and is normally sent to just one neighbour. A node "SHOULD send periodic (scheduled) Multicast Hellos unless neighbour discovery is performed by means outside of the Babel protocol" — i.e. the RFC explicitly permits skipping the Hello-based discovery path entirely if the link layer already provides discovery (as BLE GATT/advertising does).
15. **IHU TLV fields** (§4.6.6): `AE` (address encoding), `Rxcost` (the sender's own reception cost for that link, used by the *receiver* to compute its txcost), `Interval` (upper bound in centiseconds for next IHU; MUST NOT be 0), `Address` (destination address, uncompressed). IHUs "MAY be sent less often than Hellos... in particular, they SHOULD be sent less often than Hellos over links with little packet loss," and while "conceptually unicast," may be sent to multicast for aggregation/ARP avoidance.
16. **Cost computation is a two-sided exchange**: rxcost is derived locally from Hello reception history (Appendix A.1/A.2) and sent to the neighbour in an IHU; the neighbour stores that value as its *txcost* for the link and combines rxcost+txcost into the link cost, per §3.4.3, subject only to three MUST conditions: cost is strictly positive; cost is infinite if no Hellos received recently; cost is infinite if txcost is infinite. "How the txcost and rxcost are combined... is a matter of local policy."
17. **Liveness timers are Hello/IHU-driven, not connection-driven**: a neighbour's IHU timer is "set to a small multiple of the interval carried in IHU TLVs"; when it expires, "the neighbour's txcost is set to infinity" (§3.4.2) — this whole liveness model assumes a *link that persists between periodic Hello/IHU exchanges*, i.e. it detects silence-over-a-live-link, not a link that itself connects/disconnects at the transport layer as BLE GATT does.
18. **MeshLink's current wire fields for Hello/IHU are a reduced subset of the RFC's fields.** Per `RoutingPayloadCodec.kt`/`WirePayloadCodecSchema.kt`: `WireFrame.Hello` carries only `peerId` and `helloIntervalMillis` — there is **no Hello seqno field at all** (no Multicast/Unicast Hello counter, no Unicast flag), so MeshLink cannot currently distinguish scheduled vs. unscheduled Hellos or detect missed/reordered Hellos the way RFC 8966's Hello-history algorithm (Appendix A.1) requires. `WireFrame.Ihu` carries only `peerId` and `receiveCost` — there is **no IHU interval field**, so a receiver cannot compute an IHU hold-timer the way §3.4.2 specifies ("resets the IHU timer... to a small multiple of the interval value received in the IHU"). Both of these are wire-format gaps relative to RFC 8966, not just missing coordinator logic — reproducing the RFC's Hello-history/IHU-hold-timer algorithms as specified would require adding fields (`interval` to Hello alongside `helloIntervalMillis` semantics reconciliation, and an explicit IHU interval/hold-time field distinct from `receiveCost`).

### 4. RFC 8966 has no route-table digest / differential-update mechanism (Section 3.7 vs. whole document)

19. RFC 8966's only "keep traffic small" mechanisms are: (a) periodic *full* per-prefix Update TLVs on an Update interval (§3.7.1, "every Babel speaker MUST advertise each of its selected routes... at least once every Update interval"), (b) event-triggered incremental Updates for individual prefixes (§3.7.2, "Triggered Updates" — sent per-prefix on router-id change, retraction, or significant metric change), and (c) TLV/packet-level compression tricks for encoding many individual Update TLVs compactly (prefix compression, router-id/next-hop TLV state reuse — §4.5). None of these are a *table-level digest/hash comparison with differential resync*; the RFC's convergence model is "keep sending small per-prefix deltas reliably enough, forever," not "compare table hashes and resync the difference."
20. A `grep`-level read of the full RFC table of contents (Sections 1–7, Appendices A–F) shows no TLV or sub-TLV type for anything resembling a table checksum, table version vector, or bulk differential-sync request beyond the wildcard "full route table dump" answer to a wildcard Route Request (§3.8.1.1: "When a node receives a wildcard route request, it SHOULD send a full route table dump. Full route dumps SHOULD be rate-limited"). That is the closest RFC 8966 analogue to a "resync," and it is a full dump, not a differential/digest-based one.
21. **Conclusion: MeshLink's `RouteDigest` concept has no RFC 8966 precedent.** It is an original MeshLink design decision that needs its own justification (bandwidth/BLE-MTU rationale, false-positive/negative behavior under a 32-bit hash, what triggers sending it, what a receiver is supposed to do on mismatch) rather than being described as "implementing what RFC 8966 says." This matters directly for issue #130's design step: whatever RouteDigest resync semantics get specified will be original protocol design, not RFC-compliance work.

### 5. MeshLink's current implementation, grounded in code

22. **`directRouteSeqNos` reconnect-driven bump** — `RouteCoordinator.onPeerConnected` (meshlink/src/commonMain/kotlin/ch/trancee/meshlink/routing/RouteCoordinator.kt:31-38) unconditionally computes `val seqNo = (directRouteSeqNos[peerId.value] ?: 0L) + 1L` and stores it every time a peer connects — this is the sole seqno-minting path found in the reviewed routing sources; there is no separate "seqno request satisfied → increment by 1" path implemented, and no path where a *non-owning* node's stored seqno for a destination increases except by copying a received `RouteUpdate.seqNo` value verbatim (`onRouteUpdate`, same file, sets `seqNo = update.seqNo` into the new candidate route unmodified).
23. **`RouteDigestTracker`** (meshlink/src/commonMain/kotlin/ch/trancee/meshlink/routing/RouteDigestTracker.kt) maintains a sorted-by-destination-peer-id concatenation of per-route encoded bytes (`destinationPeerId`, `nextHopPeerId`, `metric`, `seqNo` truncated to `Int`, both public keys) and reduces the whole thing to a single 4-byte FNV-1a hash (`fnv1a32`). It exposes `routeDigestFrame(localPeerId)` which is attached to essentially every advertisement fan-out in `RouteAdvertisementPlanner` (peer-connected, peer-disconnected, route-update, route-retraction cases all call `addFrameAndDigest`/`addDigest`), i.e. the digest is broadcast opportunistically alongside every routing mutation, but there is **no logic anywhere in the reviewed files that compares a received digest against the local one or requests anything as a consequence of a mismatch.**
24. **`RouteCoordinator.onRouteDigest` is a literal no-op**: `@Suppress("UNUSED_PARAMETER") internal fun onRouteDigest(fromPeerId: PeerId, frame: WireFrame.RouteDigest): Unit {}` (RouteCoordinator.kt, near the bottom of the class). The receive side of the RouteDigest protocol does nothing at all today — no comparison, no triggering of a resync/route-request, no diagnostics. This is the central gap issue #130 needs to design around: RouteDigest is currently "broadcast a table hash" with zero consumption logic.
25. **`WireFrame.SeqNoRequest` exists on the wire** (`destinationPeerId`, `requestedSeqNo`, per `RoutingPayloadCodec.kt`) but `RouteCoordinator` has no handler function for it among the reviewed `on*` methods (`onPeerConnected`, `onPeerDisconnected`, `clearConnectedPeers`, `onRouteUpdate`, `onRouteRetraction`, `onRouteDigest`, `nextHopFor`, `routeFor`) — i.e. nothing in the coordinator currently originates, forwards, or answers a SeqNo request, and (per Finding 11) the frame lacks a hop-count field needed for RFC-style loop-safe forwarding.
26. **`docs/explanation/understanding-babel-routing.md`** (MeshLink's own account of its design) already flags this exact tension: "Reconnect logic is not only about 'the peer is back.' The routing state also has to look fresh enough for neighbors to re-accept and re-propagate it. If reconnect uses stale sequence information, the local node may look healthy again while neighboring nodes keep older route state and fail to spread the recovery. That is why reconnect paths need a fresh seqno progression." — This is the doc's own justification for the current reconnect-driven bump, and it is explicitly framed as a deviation necessitated by BLE reconnect churn, not as RFC-compliant behavior. The doc does not mention seqno-request/starvation-recovery at all, nor differential RouteDigest semantics beyond "MeshLink does not dump the full route table on every change... sends the route changes that matter" (which describes triggered-update behavior, not the digest hash mechanism actually implemented in code).
27. **`docs/rfcs/routing/m009-routing-metadata-privacy.md`** documents a *different*, apparently earlier/parallel wire encoding for route control frames — `BabelRouteFrameCodec` with `UPDATE = 0x21` / `WITHDRAWAL = 0x22`, plus new `ROUTE_CAPS = 0x20` and `ROUTE_PRIVACY_ENVELOPE = 0x23` frame types for an encrypted-envelope negotiation layer — that does not obviously line up 1:1 with the `WireEnvelopeType`/`RoutingPayloadCodec` (`HELLO`, `IHU`, `ROUTE_UPDATE`, `ROUTE_RETRACTION`, `SEQNO_REQUEST`, `ROUTE_DIGEST`) codec reviewed above. M009 does not mention Hello/IHU, SeqNoRequest, or RouteDigest frame types at all; it is scoped narrowly to UPDATE/WITHDRAWAL privacy-envelope negotiation. Any Hello/IHU/SeqnoRequest/RouteDigest wire-format work done for issue #130 will need to reconcile with (or explicitly scope around) whatever privacy-envelope negotiation model M009 established, since M009's envelope only wraps `0x21`/`0x22` inner frames today.
28. **`specs/ble-mesh-sdk/spec.md`** (reviewed first ~120 lines) states a hard product constraint relevant to any redesign: control-plane route convergence must complete within a 3-second budget after a topology change in a 10-node topology over "the canonical virtual transport" (User Story 2's acceptance scenario). This is a concrete latency budget that any seqno-request-forwarding-based (destination round-trip) design has to fit inside, given BLE GATT/L2CAP hop latency. `specs/ble-mesh-sdk/data-model.md` was not found with that exact path during this pass (not independently verified in this research session) — recommend confirming/relocating it in the design step.

### 6. How other churny-link/BLE-mesh routing schemes handle freshness and liveness without stable-link assumptions

29. **B.A.T.M.A.N. / batman-adv** (a flooding distance-vector protocol explicitly designed for ad hoc wireless mesh) uses a *locally-minted, periodically-incremented* originator sequence number rather than a destination-round-trip request model: each node broadcasts an Originator Message (OGM) at a fixed interval containing its own address and an ever-increasing sequence number; neighbours maintain a fixed-size *sliding window* of recently-seen sequence numbers per originator and compute "Transmission Quality" as the fraction of expected OGMs actually received in that window — link/route quality is inferred purely from OGM-arrival density over a rolling window, not from an explicit Hello/IHU cost exchange. ([BATMAN Adv. mesh network emulator, sedici.unlp.edu.ar](http://sedici.unlp.edu.ar/bitstream/handle/10915/50450/Documento_completo..pdf-PDFA.pdf?sequence=1); [B.A.T.M.A.N Unpacked, TUM](https://www.net.in.tum.de/fileadmin/TUM/NET/NET-2024-09-1/NET-2024-09-1_02.pdf))
30. batman-adv explicitly avoids any dependency on synchronized clocks or a live round-trip to the originator to refresh freshness — freshness is entirely relative-counter-based and self-refreshing on a fixed period, which sidesteps the "no live round trip to fetch a fresh seqno" problem Babel's seqno-request model has, at the cost of "spontaneous seqno bump" behavior that RFC 8966 explicitly discourages for its own feasibility-condition reasons (Finding 1). ("BATMAN-adv relies on periodic OGMs with relative sequence numbers... ensuring that routing remains robust even when nodes have no synchronized clocks, reboot frequently, or move rapidly." — [Simulation of the availability and initialization speed of a layer 2 mesh, ceur-ws.org](https://ceur-ws.org/Vol-4145/paper4.pdf))
31. batman-adv's bidirectional-link confirmation is itself a self-referential freshness check rather than a stable-link Hello/IHU exchange: a node checks whether a received OGM claiming to originate from itself falls within a bounded sequence-number window (`BI_LINK_TIMEOUT`) of its own current counter, to confirm the link back is live and roughly synchronized, rather than relying on a persistent per-neighbour Hello timer. ([B.A.T.M.A.N Unpacked, TUM](https://www.net.in.tum.de/fileadmin/TUM/NET/NET-2024-09-1/NET-2024-09-1_02.pdf))
32. **Bluetooth Mesh's own routing model (managed flooding) sidesteps Babel-style route computation entirely**: there is no distance-vector table exchange at all. Every node keeps a strictly increasing local `SEQ` (sequence number) per source address, used purely for *replay protection*, not for route freshness/feasibility — "Nodes will reject any message with a sequence number less than the last properly received good message sequence number." Reachability across intermittent/mobile links is handled by TTL-bounded flooding plus a message cache for duplicate suppression, not by neighbour-table liveness timers: "a message is only retransmitted when its TTL is greater than one... a received message that already exists in the cache is automatically discarded and not retransmitted." ([Silicon Labs, Bluetooth Mesh Networking Principles](https://docs.silabs.com/btmesh/latest/btmesh-network-optimization/03-bluetooth-mesh-networking-principles); [Hernández Solana et al., "Bluetooth Mesh Analysis, Issues, and Challenges," IEEE Access 2020](https://upcommons.upc.edu/bitstream/handle/2117/339663/09035389.pdf))
33. Bluetooth Mesh's Heartbeat messages are the closest analogue to a liveness/topology-distance signal, but they are optional, coarse (report hop-count distance, not link cost), and explicitly used only to help set TTL, not to run a Bellman-Ford-style route computation: "Heartbeat messages, transmitted by devices at intervals, include information which allows the network to learn about its topology and the number of hops away each of the other devices is." ([Bluetooth SIG, "An intro to Bluetooth Mesh Networking, part 2"](https://www.bluetooth.com/blog/an-intro-to-bluetooth-mesh-part2))
34. **General DTN/opportunistic-routing literature** treats "no live round trip / frequent disconnection" as the defining problem of the domain and answers it with store-carry-forward epidemic/probabilistic routing (e.g., PRoPHET's delivery-predictability metric built from encounter history) rather than any seqno-freshness patch to a distance-vector protocol — i.e., the DTN literature's answer to "assume the link isn't stable" is a different routing paradigm altogether, not a Babel variant. This is useful context (it shows the problem space is well studied) but does not offer a drop-in Babel-compatible fix. ([An Improved PRoPHET Routing Protocol in Delay Tolerant Network, PMC](https://pmc.ncbi.nlm.nih.gov/articles/PMC4306380))
35. No source found in this pass describes a published "Babel over BLE" or "Babel over frequently-disconnecting links" adaptation specifically; the closest analogues are batman-adv (destination-sourced-but-periodic seqno + sliding-window liveness, no round-trip request) and Bluetooth Mesh's own managed-flooding model (no distance-vector table at all, freshness solved purely via monotonic per-source SEQ + replay cache + TTL). Both avoid Babel's request/response round-trip to the destination for freshness, which is the crux of what issue #130 needs to resolve for MeshLink.
