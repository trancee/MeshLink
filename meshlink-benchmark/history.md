## Current release-decision posture

The released baseline still carries the documented iOS waiver history, but the latest current-HEAD future branch now has retained recipient-confirmed evidence that clears `SC-004` on both reference peers. The earlier post-remediation recipient-confirmed 64 KiB series restored clean physical proof completion on both reference peers—OPPO finished 5/5 at `14.50-17.09 KB/s` (avg `15.57 KB/s`) and Samsung finished 5/5 at `27.85-33.56 KB/s` (avg `30.64 KB/s`)—and a later deeper cross-platform role-policy redesign deterministically routed mixed Android/iOS peers through the faster inbound/iPhone-peripheral-hosted path. Final retained clean-build evidence on that deterministic path reached `52.03 KB/s` on OPPO and `39.48 KB/s` on Samsung. Fresh current-HEAD inline mixed-bearer reruns first split the matrix (`66.12-71.67 KB/s` OPPO versus `52.72-58.99 KB/s` Samsung), then a later discovery-suspension remediation lifted the same product path into the `61.13-68.67 KB/s` class on Samsung and the `77.02-78.24 KB/s` class on OPPO, both with passive proof receipts retained on attempt 1. A final proof-app auto-send task de-duplication fix then removed a duplicate-benchmark false negative after transient peer-loss / route-recovery sequences and retained an immediate 5-run Samsung series at `60.38-63.81 KB/s` (avg `62.41 KB/s`) plus an immediate 3-run OPPO series at `74.51-79.50 KB/s` (avg `77.20 KB/s`). ## Current learnings

- MeshLink direct L2CAP links should remain unpaired; the app-layer Noise and TOFU model already provides the intended security boundary.
- Android API 36 behaved best when explicit insecure LE socket settings were preferred first and legacy insecure LE CoC APIs were kept as fallback for older releases.
- iOS must honor the deterministic initiator tie-break. When iOS loses the key-hash ordering decision, it should wait for the inbound L2CAP channel instead of forcing a competing outbound reconnect.
- Disabling the iOS large-inline send path improved large-payload reliability, but it did not solve throughput.
- The queued-writer follow-up (`c45b7dc`) did not materially change the iPhone 15 throughput class: the latest clean OPPO rerun reached `20.05 KB/s` and the latest telemetry-enabled Samsung rerun reached `19.62 KB/s`, still roughly 3× below the `>= 60 KB/s` target.
- A later bounded conformance-path experiment widened the iOS inner write-batch size to `16 KiB` while keeping coalesced writes at `8144` bytes. That removed the extra inner batch boundary (`writeBatches=1` instead of `2`) but still measured only `18.37 KB/s` on a fresh iPhone 15 -> OPPO transient-appId rerun, so the change was reverted. This strengthens the earlier conclusion that CoreBluetooth stream availability / OS scheduling dominates more than the inner 4 KiB batching boundary itself.
- Additional 2026-05-14 Samsung follow-up reruns also failed to beat the existing `33.56 KB/s` best case. Stream run-loop scheduling (`27.33 KB/s`), shorter sender ACK-settlement timers (`24.90 KB/s`), a 32-frame transport coalescing window (`28.33 KB/s`), a re-enabled 64 KiB inline path (`31.84 KB/s`), and the same inline path with a 16 KiB inner batch (`23.05 KB/s`) all remained below the then-current best retained Samsung series. The strongest negative evidence came from the inline path: even a single `66037`-byte coalesced write still spent `1471 ms` in the CoreBluetooth output loop with `readyFalseCount=1100`, so the blocker is no longer just transfer-chunk bookkeeping.
- A more invasive iOS-only follow-up that requests `CBPeripheralManagerConnectionLatencyLow` on inbound L2CAP channels improved the pre-redesign Samsung best case to `38.28 KB/s`, but it was still role-dependent until the later mixed-platform initiator redesign.
- The deeper cross-platform role-policy redesign now uses discovery-platform hints to make the faster inbound/iPhone-peripheral-hosted path deterministic on mixed Android/iOS peers. Final clean-build retained evidence reached `39.48 KB/s` on Samsung and `52.03 KB/s` on OPPO, with the low-latency request present on every retained final-matrix run.
- Additional post-role-policy experiments still failed to beat that deterministic baseline: shrinking the iOS coalescing window to 8 frames, re-running the deterministic path with the 64 KiB inline MeshLink path re-enabled, and raising the inbound ACK batch threshold to 32 all landed back in the ~31-37 KB/s class.
- Combining the low-latency request with the re-enabled 64 KiB inline MeshLink path still did not help: the best such run after the deterministic role-policy change reached only `36.99 KB/s` on OPPO and `35.50 KB/s` on Samsung.
- The remaining iOS `SC-004` non-conformance now looks increasingly platform-limited: on the current public Android `BluetoothSocket` / LE L2CAP APIs and iOS Core Bluetooth L2CAP APIs, MeshLink has exhausted the obvious app-layer initiator, batching, and ACK-cadence levers without reaching `>= 60 KB/s`.
- The fresh telemetry-enabled Samsung rerun no longer stalls before a terminal benchmark line, which suggests the queued writer keeps the app-side pipeline fed. Waiting for full iOS notify-frame drain before reporting delivery, serializing Android GATT-notify decode/order handling, and capping Android->iOS GATT-write chunks to a safer 512 bytes restored recipient-confirmed 64 KiB product-path completion on Samsung (`45.52 KB/s`) and OPPO (`52.55 KB/s`). - A later proof-app harness fix also removed a false-negative warmup-only rerun. Without that guard, one transient Samsung rerun emitted two warmups and two scored benchmark sends for the same physical run. After the de-duplication fix, fresh immediate retained series stayed above target on both reference peers: Samsung `60.38-63.81 KB/s` across 5 runs (avg `62.41 KB/s`) and OPPO `74.51-79.50 KB/s` across 3 runs (avg `77.20 KB/s`).

Advancing sender/session state on queue acceptance was enough to desynchronize real encrypted traffic under load.
- Android central implementations cannot assume `onCharacteristicChanged` callbacks arrive serially on one thread. At least one reference OEM path delivered notifications concurrently, so notification bytes must be snapshotted immediately and framed/decrypted through an ordered pipeline.
- Safe Android->iOS GATT side-link writes are more conservative than the negotiated `mtu=517` alone suggests. The retained Samsung receipt failure narrowed to repeated `status=9` write failures for a 533-byte framed payload; chunking and capping writes to 512 bytes restored receipt-path closure.
- Headless `devicectl` proof captures must use a hard timeout plus non-blocking or idle-aware stdout handling. The earlier ad-hoc runner appeared to hang only because it waited forever for another console line after the scored benchmark result, while the proof app remained alive but quiet.

At least one retained path required temporary inbound L2CAP links to be promoted onto the later resolved discovered peer ID before the real peer could use the accepted channel reliably.

That points more strongly to a return-path session / route collapse after payload receipt than to a sender-side benchmark-format bug.

Three other post-remediation matrix cells (Samsung 64 KiB, OPPO 256 B, OPPO 64 KiB) failed earlier as sender-side `NotSent(reason=UNREACHABLE)`. The helper therefore improved diagnosis, not conformance.

That points to a shared post-payload peer-reappearance failure, plus an extra stale-route / stale-presence asymmetry on the Samsung path.

- The recipient-confirmed 64 KiB proof-completion blocker is now closed on the current reference path: the post-remediation OPPO series finished 5/5 at `14.50-17.09 KB/s`, and the post-remediation Samsung series finished 5/5 at `27.85-33.56 KB/s`.
- The remaining physical performance blocker is iPhone 15 large-transfer throughput, not discovery, trust pinning, pairing, or recipient-confirmed completion stability on the current MeshLink path.
- Android receipt reliability had two distinct, real code-level root causes rather than one: (1) unconditional per-scan-result verbose logging on the main thread was slow enough to starve BLE handshake work under fleet load, and (2) the battery-driven power-policy updater called the full scan/advertise restart path on every battery snapshot regardless of whether the resolved `PowerTier` actually changed, which self-inflicted Android's silent BLE scan throttle. Both are now fixed and log-verified (commits `2caf37da`, `90b8d75b`, `9feda44c`).
- Not every observed BLE flakiness is a code bug. A rigorous A/B protocol (clean device reboot to clear accumulated throttle/radio state, then an isolated Bluetooth-adapter-only power cycle) is the way to distinguish a genuine app-level defect from a device-specific chipset/firmware wedge: the wedge reproduced even on the very first run after a full reboot and was tied to device identity, not to launch order, sender/receiver role, or extras.
- A generation-tagged scan watchdog (stop/start the scanner after a bounded no-results idle window, well above Android's undocumented restart-throttle window) is a reasonable defense-in-depth mitigation for wedged scanners, but it cannot fully substitute for an actual Bluetooth adapter power-cycle; some device-specific wedges survive several watchdog-triggered `stopScan()`/`startScan()` cycles with zero results delivered, and only clear on a full adapter reset that ordinary app code cannot invoke without elevated permissions.
- Retained best-case throughput headlines must be periodically re-verified against the current fleet ledger, not just carried forward. The previously retained `1306.12 KB/s` Android figure did not correspond to any run in the current ledger and, once re-measured with the standard reference pairing, the real number was ~130x lower (`8.47-10.36 KB/s`). A single un-repeated best-case sample is not a reliable regression baseline; prefer a small repeated series (as already used for the iOS matrices) for any headline throughput figure.
- Android throughput is bearer-dependent, not just chipset-dependent: `BEARER decision` diagnostics showing `mode=GATT_ONLY` or `mode=L2CAP_PREFERRED_WITH_GATT_FALLBACK l2capReady=false` reveal when a pairing silently fell back to the slower GATT bearer. The Samsung Galaxy XCover 4 (SDK 28 / Android 9) cannot host an L2CAP server socket at all (`BluetoothServerSocket` L2CAP CoC listener APIs need API 29+), so any pairing involving it as receiver is permanently capped at GATT-fallback throughput regardless of the sender's own L2CAP capability. This is a per-device SDK/hardware ceiling that a future benchmark baseline should track per-pairing rather than as one fleet-wide number.
- The GATT-fallback bearer's chunk write path (`GattNotifyClient`) was fully stop-and-wait: `WRITE_TYPE_DEFAULT` plus awaiting `onCharacteristicWrite` before issuing the next 512-byte chunk, capping throughput at roughly one chunk per BLE connection-interval round trip and matching the observed 8-10 KB/s exactly. Switching to `WRITE_TYPE_NO_RESPONSE` with a bounded 4-chunk write-ahead pipeline (a FIFO of pending write acks drained via a semaphore-gated window) removes that stop-and-wait ceiling.
- Naively pipelining `WRITE_TYPE_NO_RESPONSE` writes surfaces a real Android BLE quirk: `BluetoothGatt.writeCharacteristic()` can transiently return `false` (an immediate enqueue failure) right after a previous write's completion callback fires, even though nothing is actually wrong with the link — the framework's internal busy flag / ACL buffer credit visibly lags the app-facing completion callback, and it is more pronounced under `WRITE_TYPE_NO_RESPONSE` than the old write-with-response path. The existing higher-level logic treats any chunk write failure as fatal and tears down/reconnects the whole GATT side-link, so an unguarded pipeline change regressed physical hardware to full transfer timeouts before a fix was added.
- The correct fix is bounded retry-with-backoff at the enqueue call site (5 attempts, 20ms apart) rather than immediate failure, treating the transient `false` return as backpressure instead of a fatal error — matching community-documented practice for Android GATT "busy" responses. After adding this retry, a fresh physical fleet rerun (Samsung Galaxy XCover 4 <-> OPPO Reno8 5G, 64 KiB payload, 4 repeats) showed no more enqueue-failure-triggered reconnect loops, and per-direction throughput rose from the 8.47-10.36 KB/s GATT-fallback baseline to 39.78-47.16 KB/s (Samsung-as-sender) and 17.03-18.11 KB/s (OPPO-as-sender) — a 2-5x improvement, though still short of the 80 KB/s SC-004 target and still asymmetric by sender role, which likely reflects remaining connection-interval/central-vs-peripheral role asymmetry rather than the write path itself.
- Two of the eight runs in that same rerun timed out entirely with no scored benchmark line, both with the Samsung device as sender. Their sender logs show the GATT side-link reaching `ready` normally with no enqueue-failure symptoms at all, then the higher-level route/session layer never reporting readiness — this pattern (side-link connects fine, route readiness silently never arrives) matches the already-documented residual device-specific BLE scan wedge from the receipt-reliability investigation, not a new regression introduced by the write pipelining change.
- The scan watchdog's plain scan-restart mitigation was insufficient for the residual wedge: it can restart the scanner but not clear whatever internal chipset/firmware state actually causes the wedge, so the wedge can survive many consecutive scan restarts. Escalating after `MAX_WEDGED_SCAN_RESTARTS_BEFORE_ESCALATION` (2) consecutive restarts that fail to bring in a scan result -- first attempting a full `BluetoothAdapter.disable()`/`enable()` power-cycle, then falling back to a `DiagnosticCode.MANUAL_BLUETOOTH_RECOVERY_NEEDED` signal when the power-cycle can't even be attempted -- was validated end-to-end on physical hardware (Samsung Galaxy XCover 4 <-> OPPO Reno8 5G): the OPPO Reno8 5G's receiver-side log for a timed-out run showed `scan watchdog restarting scan ... consecutiveWedgedScanRestarts=1`, then `=2`, then `scan watchdog escalating wedge recovery powerCycleAttempted=false`, then two `DIAG MANUAL_BLUETOOTH_RECOVERY_NEEDED` diagnostics roughly a minute apart -- confirming the OPPO device's own scanner (not the Samsung sender) never discovered the peer at all in that run, exactly matching the previously-documented wedge.
- `BluetoothAdapter.disable()`/`enable()` only actually power-cycle the radio for a non-privileged app on Android 12 (API 32) and below; Android 13+ (API 33+, TIRAMISU) made both methods no-ops without `BLUETOOTH_PRIVILEGED` (system-app-only) or `DevicePolicyManager` device-owner status. The OPPO Reno8 5G reference device runs API 34, so on that device the escalation path always falls back to the manual-recovery diagnostic rather than an actual adapter reset -- a real user physically toggling Bluetooth remains the only way to clear the wedge on that specific device/OS combination. The escalation logic is still worthwhile: it will actually power-cycle the radio on older/lower-API devices (e.g. the Samsung Galaxy XCover 4, API 28), and on any device it turns a silent, indefinite hang into an actionable diagnostic the host app can surface to the user.
- A same-day 5-device fleet investigation (2026-07-11) confirmed the existing peer-flap hysteresis already works as designed under real RF congestion: the GATT side-link's 4-second `PEER_LOST_DEBOUNCE_MILLIS` genuinely waited out a transient disconnect before reporting `PeerLost`, and `L2capReconnectGuard`'s 3-attempt/500-1000-2000ms backoff schedule is gated on the disconnect reason string rather than firing unconditionally. Both mechanisms already carry dedicated regression coverage (`GattSideLinkCoordinatorTest`'s PeerLost-debounce assertions, `L2capReconnectGuardTest`'s backoff-schedule assertions), so this investigation's live-hardware observation corroborates existing, already-tested behavior rather than being the only evidence for it; neither mechanism needed a code change. The pair-scoped congestion failures observed instead traced to (a) genuine BLE packet-collision `AEADBadTagException` bursts that recovered on their own without desyncing the session, and (b) mid-run BLE MAC address rotation on one peer forcing a real GATT reconnect that itself hit a transient `GATT_CONNECTION_TIMEOUT` before succeeding ~8.8s later -- a compounding-latency case, not a missing-debounce bug.
- The MeshLink app-layer peer identity (key-hash-based, carried in the BLE discovery payload) correctly survived a mid-run BLE MAC address rotation on a live peer: the same `PeerId` was recognized as "found" again under the new MAC once the old GATT link was locally terminated, with no duplicate-peer or identity-confusion symptoms. MAC rotation only affects the transport-level link, not mesh routing/session identity.
- The initial 3-repeat `--full-mesh` finding that "the two oldest/lowest-spec devices never confirm anything under N-way load" turned out to be an overfit to a small sample once a controlled fleet-size sweep (2/3/4/5 devices, same physical units) was run the same day: those two devices actually succeeded fully (6/6) once tested at 3-device scale, including full round trips with the exact two peers each had just failed against in 2-device isolation moments earlier. The real, reproducible signal was narrower and sharper -- a threshold specifically at the full 5-device fleet size, driven by two separate per-device BLE stack quirks (an intermittent scan-result wedge reproduced on two different devices' scanners, and a Samsung unit's `l2capServerSupported=false` GATT-only ceiling) compounding together, not a smooth N-way negotiation-capacity curve. Treat any single small-repeat-count fleet finding as provisional until a fleet-size (or other single-variable) sweep confirms the shape of the effect, not just its presence at one tested size.
- `PowerPolicy.maxConnections` (7/5/3 per tier) was, until this investigation, computed and read only into diagnostics -- nothing in the codebase actually admission-controlled a new BLE connection against it. A shared, platform-agnostic `hasConnectionBudget()` decision function (`power/PowerPolicy.kt`) plus Android wiring in `BleTransportAdapterScanSupport`'s scan-result handler now defers starting a new GATT side-link/L2CAP connection for a not-yet-connected peer once the device's active connection count (GATT side-links union L2CAP links) reaches the tier's budget; an already-connected peer is always still admitted so it never gets forcibly dropped. Confirmed via a live 5-device rerun that this is a true no-op at the fleet sizes tested here (well under the 7-connection PERFORMANCE-tier budget, no "connection budget exhausted" log lines, identical 4/20 confirmed-pairs result to the pre-change baseline) -- this closes a real admission-control gap but is not expected to change today's 5-device fleet results, since none of the observed failures were connection-count-related (see the scan-wedge and L2CAP-capability findings above). Only wired into Android in this change; iOS enforcement is an explicit follow-up, since this session had no iOS build/simulator environment to validate against (commonMain compilation for iosArm64/iosSimulatorArm64 was confirmed clean via `apiCheck`/`allTests`, but no iOS runtime test was run).
- A follow-up targeted investigation (same day) root-caused why Huawei WAS-LX1A specifically never completed a confirmed round trip in any tested configuration despite substantial peer interaction: the GATT-notify side link's `LinkIdentity` announcement was piggybacked on the *first real payload write*, which a pure Noise-XX handshake *responder* legitimately never makes (it waits for the initiator's message1). Combined with the peer's own scanner never independently discovering the responder (the scan-wedge phenomenon documented above), the peer had no way to ever learn the responder's identity, and the pairing deadlocked permanently -- fully connected at the GATT layer, zero further protocol activity, forever. Fixed by proactively announcing the identity as soon as the link becomes ready (`GattNotifyClient.ensureIdentityAnnounced()`, called from `onDescriptorWrite`), independent of whether this device ever has a real payload to write. Live-hardware validation before/after: isolated Huawei+Xiaomi pairs went from 1/2 to 6/6 confirmed across repeats; isolated Huawei+Samsung pairs went from 0/2 to 3/3 confirmed. A full 5-device rerun after the fix still showed the separate, already-documented RF-congestion failure mode at that fleet size (`4/20`, real logged handshake-send *attempts* that then failed under load) -- this fix closes the silent no-attempt deadlock specifically, not the N-way congestion behavior.
- Post-merge fresh-build confirmation (same day, all three fixes -- `--full-mesh` mode, `maxConnections` enforcement, and the identity-announce fix -- combined on `main`): a fresh APK build, uninstall-then-reinstall across all 5 devices, and 3 repeated full 5-device `--full-mesh` reruns scored `6/20`, `8/20`, `3/20`. The `8/20` run is the first time in this entire investigation that Huawei WAS-LX1A appears in the confirmed matrix at full 5-device fleet scale at all -- it confirmed both directions with Samsung SM-G970U1. WAS-LX1A's own logcat for every rerun now shows `GATT notify side link ... announced local LinkIdentity=...` firing immediately upon link readiness for every discovered peer (the previously-silent deadlock symptom is gone), with remaining non-confirmations at this fleet size showing real, logged `BEARER decision`/`GATT side-link send attempt`/`HOP_SESSION_*` activity that then fails under load -- i.e. genuine RF congestion, never the silent no-attempt deadlock again. The three fixes compose cleanly with no observed regression.

- iOS `maxConnections` admission-control parity (issue #101, follow-up to the entry above): iOS's transport adapter now threads `PowerPolicy.maxConnections` through `PowerMonitor.profileFor`/`defaultProfile` into `PowerProfile.maxConnections`, and `BleTransportAdapterDiscoverySupport.handleL2capDiscovery` calls the same shared `hasConnectionBudget()` decision before initiating an outgoing L2CAP connect, using the union of active L2CAP links (`activeLinksByHint.keys`) and active GATT notify side-links (`BleTransportGattNotifyRegistry.activeHintIds()`, added alongside this change) as the active connection count -- mirroring Android's L2CAP-link/GATT-side-link union. iOS only gates the L2CAP connect because it is the only connection the iOS transport adapter initiates locally; GATT notify side-links there are inherently peer-initiated (the local device is the GATT peripheral), so there is no local iOS decision point to defer for that path, but such links still count toward the active total. This change was validated with new host (`iosSimulatorArm64Test`) unit coverage (`PowerMonitorTest`'s `maxConnections` assertions, `BleTransportGattNotifyRegistryTest`'s `activeHintIds()` tests) and `apiCheck`; no physical iOS device or CoreBluetooth-backed simulator run was exercised in this session (CoreBluetooth radio operations do not function in the iOS Simulator), so live-hardware confirmation that the admission gate behaves correctly under real fleet load remains an open validation step for a future session with physical iOS hardware available.
